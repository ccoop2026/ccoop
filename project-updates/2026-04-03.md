# Project Update — 2026-04-03

---

## Summary

### Signup — Rebuilt as a Step-by-Step Flow
- The old single-page form is gone. Signup now walks the user through **5 short screens**, one topic at a time: campus → profile → second campus → address → account details.
- Each screen only shows what's needed at that moment — far less overwhelming than one long scroll.

### Campus Selection
- Clicking the campus field opens a **real dropdown** with all registered campuses listed.
- Type to filter the list down instantly.
- If a campus isn't listed, a **"My campus is not listed"** option is always visible at the bottom.
  - Clicking it opens a panel where the user can contact the admin (email or WhatsApp) and **leave their contact to be notified** when their campus is added.

### User Profile
- Three user types supported: **Current Student**, **Alumni**, **Staff**.
- Staff gets a further split: **Teaching** or **Non-Teaching** — no passout year asked.
- Students and alumni select their **passout year from a dropdown** (no free typing), keeping the data consistent.
  - Alumni: 1974 up to this year.
  - Students: this year up to 7 years ahead.
- Department is a text field capped at **50 characters** with a live counter.

### Phone Number
- Now a **required field** — can't be skipped.
- As the user types, the app checks in the background whether the number is already registered.
- Shows a live indicator: ⏳ checking → ✓ available or ✕ already registered — before they even hit submit.

### Home Page (after login)
- Greets the user by name and shows their **linked campuses**.
- Two clear entry points: **Active Campaigns** and **Store**.
- If a channel isn't live yet, the user sees a friendly "not active yet — we'll notify you" screen instead of a blank page.

### Campaigns Page
- Shows live merchandise drops for the user's registered campus.
- If two campuses are linked, **tabs let the user switch** between them.
- Cart is one item at a time. Switching campus with an item in the cart shows a prompt — exit cart or stay.

## Client Feedback & Backlog

### High Priority
- **Remove the username field from signup.** Users should be identified by email or phone — no separate username needed.
- **Refine the address field format.** The current single text box isn't structured enough for logistics. Should be broken into proper fields: flat/hostel no., building, locality, city, pincode.

### Medium Priority
- **Decide when to collect the address.** It's currently asked during signup (Step 4). The question is whether it should move to cart checkout instead, or appear in both places. Needs a product call before changing.

### Low Priority
- **OTP verification for phone and email.** Users should verify ownership of their number/email during signup. Deferred — needs an SMS and email gateway to be set up first.
- **Gmail / Google sign-up.** The button is already on screen but it doesn't do anything yet. Backend integration pending.

## Known Gaps (not backlog — just current state)

- Registration does not yet write to the database. The token stored in the browser is a placeholder. Real user creation and login are not wired to Django yet.
- The phone uniqueness check does hit the real database, but since no users are created through the UI yet it will always return available.
- Cart is browser-only (`localStorage`). No backend order model exists yet.
- Campus and session data are read from `localStorage` keys (`user_campuses`, `auth_token`, `username`) set at the end of the registration flow.
